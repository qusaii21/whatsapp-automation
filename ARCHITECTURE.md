## 3. Table Relationships

```
leads (1) ──────────────── (∞) conversations
leads (1) ──────────────── (∞) lead_notes
leads (1) ──────────────── (∞) followups
leads (1) ──────────────── (∞) ai_classifications

conversations (1) ──────── (∞) messages
conversations (1) ──────── (∞) ai_classifications

messages (1) ────────────── (0..1) ai_classifications

webhook_events ─────────── independent log table
                            (idempotency_key is UNIQUE)
```

### Entity Relationship Overview

| Table               | Depends On                     | Key Constraint                          |
|---------------------|--------------------------------|-----------------------------------------|
| `leads`             | —                              | UNIQUE (phone_e164, channel)            |
| `conversations`     | leads                          | UNIQUE (lead_id, channel, status=open)  |
| `messages`          | conversations, leads           | UNIQUE (whatsapp_message_id)            |
| `lead_notes`        | leads, agents (auth.users)     | —                                       |
| `followups`         | leads, conversations           | —                                       |
| `ai_classifications`| leads, conversations, messages | UNIQUE (message_id) partial             |
| `webhook_events`    | —                              | UNIQUE (idempotency_key)                |

## 4. Index Strategy

### Primary Access Patterns

| Query Pattern                              | Index                                              |
|--------------------------------------------|----------------------------------------------------|
| Look up lead by phone + channel            | UNIQUE (phone_e164, channel)                       |
| All conversations for a lead               | (lead_id, created_at DESC)                         |
| Open conversation for a lead on a channel  | PARTIAL on conversations WHERE status = 'open'     |
| Messages in a conversation (paginated)     | (conversation_id, created_at DESC)                 |
| Unread messages for agent                  | PARTIAL on messages WHERE read_at IS NULL          |
| Due follow-ups                             | (scheduled_at) WHERE status = 'pending'            |
| Webhook dedup check                        | UNIQUE (idempotency_key)                           |
| AI classification by lead                  | (lead_id, created_at DESC)                         |
| Unprocessed webhook events                 | PARTIAL on webhook_events WHERE processed_at IS NULL|

### Index Type Decisions
- **B-tree** for all equality/range/sort patterns (default)
- **GIN** on `leads.metadata` (JSONB) for arbitrary field searches
- **GIN** on `messages.content` tsvector for full-text search

## 5. Security Considerations

### Webhook Security
- All inbound webhooks verified via HMAC-SHA256 before any DB write
- Raw body preserved in `webhook_events` for audit/replay
- Webhook secrets stored in environment variables, never in DB

### Row-Level Security (RLS)
- All tables have RLS enabled
- Agents can only read/write leads assigned to them or their team
- `webhook_events` and `ai_classifications` are service-role-only (no direct client access)
- `admin` Supabase client (service role) only used in API routes, never in browser

### Authentication
- Supabase Auth with email/password + optional OAuth
- JWT claims carry `team_id` for multi-tenant isolation
- Short-lived access tokens, refresh token rotation enabled

### Data Protection
- Phone numbers stored in E.164 format, considered PII
- `metadata` JSONB field encrypted at rest via Supabase Vault (optional)
- No raw API keys stored in DB — only references to env vars
- Idempotency keys are SHA-256 hashes of payload content — no PII

### Input Validation
- All webhook payloads validated with Zod before DB insert
- Phone numbers normalized to E.164 via libphonenumber before storage
- Message content length enforced at DB level (CHECK constraint)

## 6. Scalability Considerations

### Horizontal Scaling
- Next.js deployed on Vercel (auto-scales, Edge Runtime for webhooks)
- Supabase handles connection pooling via PgBouncer (transaction mode)
- AI calls are async — never block the webhook response

### Database Scaling
- Partitioning `messages` by `created_at` (monthly) when > 50M rows
- `webhook_events` purged after 90 days via `pg_cron`
- Read replicas for dashboard queries when traffic grows

### Rate Limiting
- WhatsApp Cloud API: 80 messages/sec per WABA — queue-backed dispatch
- Meta webhooks: idempotency table prevents duplicate processing
- AI calls: exponential backoff + circuit breaker in `ai/router.ts`

### Multi-Channel Readiness
The `channel` enum (`whatsapp`, `instagram`, `email`, `sms`) is present on:
- `leads`
- `conversations`
- `messages`

Adding a new channel requires only:
1. A new webhook route
2. A new outbound client in `src/lib/`
3. No schema changes

### Async Processing Pattern
```
Webhook → Insert webhook_event (idempotent)
        → Call Supabase Edge Function OR pg_net HTTP call
        → Worker processes event
        → Updates processed_at
```
This decouples ingestion latency from processing latency.
